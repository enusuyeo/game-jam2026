# Gemini Lens — 게임 비주얼 분석

Gemini의 네이티브 비전 능력으로 게임 스크린샷, 에셋, UI를 분석한다.
별도 도구 설치 없이 이미지를 직접 보고 판단할 수 있다.

## 분석 모드

Director의 요청에서 모드를 자동 감지한다:

| 조건 | 모드 |
|------|------|
| "에셋", "캐릭터", "배경", "스프라이트" 언급 | `asset` |
| "UI", "메뉴", "버튼", "레이아웃", "HUD" 언급 | `ui` |
| "버그", "깨짐", "이상", "오류", "안 보임" 언급 | `debug` |
| 파일 2개 이상 + "비교", "전후" 언급 | `compare` |
| 위 어느 것도 아님 | `general` |

---

## 모드별 분석 프레임워크

### asset — 에셋 품질 검토

codeb으로 생성한 이미지/3D 렌더를 검토할 때 사용.

평가 항목:
- **컨셉 일치**: Director가 요청한 컨셉과 결과물이 일치하는가
- **스타일 일관성**: 기존 에셋들과 시각적 톤이 맞는가
- **기술 품질**: 해상도, 아티팩트, 투명도 처리, 경계선 깔끔함
- **게임 적합성**: 배경 위에 놓았을 때 잘 어울리는 크기/색감인가

출력 형식:
```
## 에셋 분석: [파일명]

**컨셉 일치**: O/X — [설명]
**스타일 일관성**: O/X — [설명]
**기술 품질**: [점수/5] — [구체적 이슈]
**게임 적합성**: O/X — [설명]

**권장 조치**: [다음 단계 또는 수정 사항]
```

### ui — UI/레이아웃 리뷰

게임 UI 스크린샷을 분석할 때 사용.

평가 항목:
- **가독성**: 텍스트가 배경 위에서 잘 읽히는가
- **터치 영역**: 버튼/인터랙티브 요소가 충분히 큰가 (최소 44x44px 권장)
- **시각 계층**: 중요 정보가 먼저 눈에 들어오는가
- **여백과 정렬**: 요소 간 간격이 일정하고 정렬이 맞는가
- **모바일 호환**: 작은 화면에서도 사용 가능한 구성인가

출력 형식:
```
## UI 분석: [화면 이름]

**가독성**: [점수/5] — [설명]
**터치 영역**: [OK/경고] — [작은 요소 목록]
**시각 계층**: [설명]
**정렬/여백**: [이슈 목록]

**개선 제안**:
1. [가장 중요한 개선]
2. [차선 개선]
```

### debug — 시각적 버그 진단

깨진 화면, 렌더링 오류 스크린샷을 분석할 때 사용.

평가 항목:
- **증상 식별**: 무엇이 잘못 보이는가 (위치, 크기, 색상, 누락)
- **원인 추정**: CSS/레이아웃/렌더링/데이터 중 어디가 문제일 가능성이 높은가
- **재현 조건**: 특정 해상도나 상태에서만 발생하는 것 같은가

출력 형식:
```
## 버그 분석: [증상 요약]

**증상**: [구체적 묘사 — 위치, 크기, 색상 등]
**추정 원인**: [가능성 높은 순서로]
**확인 방법**: [코드에서 확인할 곳]
**수정 제안**: [구체적 코드 방향]
```

### compare — 비교 분석

2개 이상의 스크린샷/에셋을 비교할 때 사용.

평가 항목:
- **차이점**: 레이아웃, 색상, 스타일, 콘텐츠 차이
- **개선/퇴보**: 어떤 면이 좋아졌고 어떤 면이 나빠졌는가
- **추천**: 어느 버전이 더 나은가, 또는 절충안

### general — 일반 시각 설명

위 카테고리에 해당하지 않을 때 사용.

보이는 모든 것을 구조적으로 설명:
- 화면 구성 요소와 배치
- 색상 팔레트와 분위기
- 텍스트 내용
- 전반적인 인상과 개선 포인트

## 절차

### Step 1: 파일 확인

Director가 제공한 이미지/스크린샷 경로를 확인한다.
파일이 존재하는지 검증:

```bash
ls -la <파일경로>
```

### Step 2: 모드 감지

Director의 요청에서 위 모드 감지 테이블을 기준으로 모드를 결정한다.
애매하면 `general`을 사용한다.

### Step 3: 분석 실행

이미지를 직접 보고 해당 모드의 프레임워크에 따라 분석한다.

### Step 4: 결과 보고

모드별 출력 형식에 맞춰 결과를 Director에게 전달한다.

**원칙**:
- **구체적으로**: "오른쪽 위", "두 번째 버튼" 등 정확한 위치 지칭
- **실행 가능하게**: 모호한 평가 대신 구체적 수정 방향 제시
- **우선순위**: 가장 중요한 이슈부터
- **간결하게**: 핵심만. 장황한 설명 금지

## 에셋 파이프라인 연계

이 스킬은 에셋 생성 파이프라인의 검증 단계에서 특히 유용하다:

```
codeb으로 이미지 생성 → [gemini-lens asset 모드로 검증] → 승인 → 다음 단계
codeb으로 3D 모델 생성 → asset-preview로 프리뷰 → [gemini-lens로 렌더 검토]
게임 실행 스크린샷 → [gemini-lens ui/debug 모드로 분석]
```

Director 확인 전에 에이전트가 먼저 시각적 품질을 판단하여 명백한 문제를 사전에 걸러낼 수 있다.
